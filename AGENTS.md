# AGENTS.md - PaperSkills 科研协作契约

> **读取声明**: Following project rules

本文档约束 AI/Agent 如何协助科研，同时规范本仓库的内容组织、验证和沉淀方式。

---

## 工作模式声明

开始任务前，Agent 必须显式声明当前工作模式：

- **`[PLANNING]`** - 规划阶段：问题界定、文献调研、方案设计
- **`[EXECUTING]`** - 执行阶段：原型实现、实验运行、数据收集
- **`[VERIFYING]`** - 验证阶段：结果检验、消融分析、误差分析
- **`[WRITING]`** - 写作阶段：论文撰写、PPT制作、汇报准备
- **`[REVIEWING]`** - 审查阶段：代码审查、论文审阅、结果复核

**默认规则**: 先做只读探索，理解上下文后再提问或行动。

## 研究背景与主线

本仓库服务于两条并行的研究主线：

### 主线 A: 大模型应用研究 (LLM Applications)
关注如何将大模型应用于具体任务，创造用户价值。

**核心关注点**:
- 任务定义与场景分析
- 系统设计与交互流程
- 效果/成本/时延/安全的权衡
- 端到端评测与真实用户价值
- 失败模式与边界案例

**典型问题**: "如何让 LLM 更好地辅助代码审查？" "RAG 系统如何平衡检索质量与响应速度？"

### 主线 B: 大模型软件工程研究 (LLM-based Software Engineering)
关注如何工程化地构建、维护和评测大模型系统。

**核心关注点**:
- Prompts/Workflows/Agents/Tool use 的系统化
- 可观测性、调试与错误追踪
- 回归测试与版本控制
- 复现性与实验管理
- 维护成本与长期演化

**典型问题**: "如何系统性地优化 prompt？" "Agent 系统的失败如何分类和追踪？"

**核心原则**: 仓库所有产出应服务于"可验证的研究贡献"，而非仅追求 demo 效果。

## 默认科研流程

Agent 必须按以下顺序协助科研工作，每步只要求最小必要产物：

### Step 1: 问题界定 (Research Brief)
**产物**: 研究简报，包含：
- 问题陈述 (1-2 句话)
- 研究主线归属 (A/B)
- 预期贡献类型 (新任务/新方法/新发现/新系统)
- 成功标准 (可量化的验收条件)

### Step 2: 文献与缺口
**产物**: 相关工作总结，包含：
- 3-5 篇核心相关论文
- 明确的研究缺口 (Gap)
- 本工作的差异化定位

### Step 3: 研究假设/贡献
**产物**: 贡献声明，包含：
- 核心假设 (可证伪)
- 具体贡献点 (1-3 条)
- 范围边界 (什么不做)

### Step 4: 实验与原型设计 (Experiment Plan)
**产物**: 实验计划，包含：
- 基线方法
- 评测指标
- 数据/场景
- 验证方式
- 消融设计

### Step 5: 实现与验证
**产物**: 可运行的原型 + 验证记录

### Step 6: 消融与误差分析 (Evaluation Summary)
**产物**: 评估总结，包含：
- 主结果对比
- 消融实验
- 错误案例分析
- 局限性讨论

### Step 7: 论文/汇报产出 (Writing Pack)
**产物**: 写作包，包含：
- 论文/PPT 草稿
- 关键图表
- 回链到实验数据的引用

## 主线 A: 大模型应用研究规则

### 必须明确 (Checklist)
- [ ] **任务场景**: 具体解决什么问题？为谁解决？
- [ ] **目标用户**: 最终使用者是谁？他们的真实需求？
- [ ] **基线方法**: 当前最佳实践是什么？
- [ ] **数据来源**: 训练/评测数据从哪来？是否代表真实分布？
- [ ] **评测指标**: 主指标是什么？辅助指标有哪些？
- [ ] **失败案例**: 系统在什么情况下会失败？
- [ ] **时延/成本**: 推理延迟和计算成本？
- [ ] **安全性**: 潜在风险 (幻觉、偏见、隐私)？

### 禁止项
- ❌ 只展示少量成功案例就宣称有效
- ❌ 没有基线对比就报告"提升"
- ❌ 只报告最好结果，隐藏失败案例
- ❌ 未控制变量就比较不同方法
- ❌ 把"工程堆料"包装成研究贡献

---

## 主线 B: 大模型软件工程研究规则

### 必须明确 (Checklist)
- [ ] **系统边界**: 系统包含什么？与外部如何交互？
- [ ] **模块职责**: 每个模块的输入/输出/职责？
- [ ] **Prompt/Tool 版本**: 所有 prompts 和工具调用是否版本化？
- [ ] **评测协议**: 如何评测？评测脚本是什么？
- [ ] **回归集**: 用于防止退化的标准测试集？
- [ ] **错误分类**: 错误如何分类和追踪？
- [ ] **复现条件**: 复现实验需要什么环境/配置/数据？

### 禁止项
- ❌ 改了 prompt 但没有版本记录
- ❌ 没有回归测试就重构核心逻辑
- ❌ 评测脚本与论文描述不一致
- ❌ 无法解释的系统行为
- ❌ 未记录的"魔法参数"

## 证据与可复现性规则

### 结论分级
所有结论必须明确标注类型：

- **`[FACT]`** - 已验证事实: 有实验数据或可靠来源支撑
- **`[HYPOTHESIS]`** - 工作假设: 待验证的推测，需说明验证计划
- **`[PLAN]`** - 后续计划: 尚未执行，需说明时间表

### 必须记录
- 模型版本与配置 (temperature, top-p 等)
- 数据切分方式 (train/val/test)
- 评测脚本与方法
- 计算成本与资源限制
- 失败案例与错误分析

### 禁止项
- ❌ 编造引用、实验结果或对比基线
- ❌ 伪造人工主观结论
- ❌ 未标注来源的"常识"陈述
- ❌ 无法回溯到原始数据的图表

---

## 写作与展示规则

### 论文/PPT 约束
- 每个结论必须能回溯到实验或可靠来源
- 未知结果必须显式标注为 `[PLACEHOLDER]` 或 `[PENDING]`
- 图表必须注明数据来源
- 禁止夸大未完成的工作
- 区分"已完成验证"和"计划中"的内容

### 引用规范
- 所有引用必须真实存在
- 引用内容必须与原文一致
- 优先引用近 5 年工作
- 平衡引用经典与最新研究

## 仓库治理规则

### Skill 组织方式
沿用 `skills/<skill-name>/` 结构：

```
skills/<skill-name>/
├── SKILL.md              # 入口文档 (必须)
├── references/           # 参考资料 (按需)
└── assets/               # 资源文件 (按需)
```

### SKILL.md 必须包含
- **触发条件**: 什么情况下使用这个 skill？
- **输入**: 需要什么输入？格式要求？
- **输出**: 产生什么输出？格式规范？
- **边界**: 什么情况下不适用？
- **参考资料**: 相关文档或资源
- **验证方式**: 如何验证输出质量？

### 禁止项
- ❌ 把一次性任务经验直接塞进 skill
- ❌ skill 没有明确的触发条件
- ❌ 无法描述验证方式的 skill
- ❌ 过于宽泛或过于狭窄的 skill 范围

## 变更注释与验证

### 需要注释的修改
对以下内容的非平凡修改，需补充 `Why / Scope / Verify`：
- 代码实现 (核心逻辑)
- Prompt 设计
- 评测配置
- 文档结构

### 注释格式
```
# Why: 解决什么问题？
# Scope: 影响范围？
# Verify: 如何验证正确性？
```

### 修改后必须
- 执行相应验证
- 汇报验证结果
- 更新相关文档

## 反模式清单

### 绝对禁止
- 🚫 **Demo 驱动结论**: 只凭几个成功案例就下结论
- 🚫 **基线缺失**: 没有对比基线就宣称"提升"
- 🚫 **选择性报告**: 只报告最好结果，隐藏失败
- 🚫 **变量失控**: 未控制变量就比较不同方法
- 🚫 **包装工程**: 把"工程堆料"包装成研究贡献
- 🚫 **编造数据**: 伪造实验结果或引用
- 🚫 **版本混乱**: 修改 prompt/配置不记录版本
- 🚫 **不可复现**: 无法提供复现条件

### 警示信号
- ⚠️ 无法解释的系统行为
- ⚠️ 评测指标与目标不一致
- ⚠️ 实验设置与论文描述不符
- ⚠️ 消融实验缺失或敷衍
- ⚠️ 错误分析流于表面

```yaml
问题陈述: str
主线归属: A | B
预期贡献: str
成功标准: List[str]
时间估算: str
```

### Experiment Plan (实验计划)
```yaml
基线方法: List[str]
评测指标: List[str]
数据集: str
验证方式: str
消融设计: str
资源需求: str
```

```yaml
主结果: Dict[str, float]
消融结果: Dict[str, Any]

局限性: List[str]
后续工作: List[str]
```

```yaml
文档类型: paper | ppt | report
核心结论: List[str]
关键图表: List[str]
数据来源: Dict[str, str]
待完成项: List[str]
```

**当前可用 Skills**:
- `research-paper-writer`: 学术论文写作 (IEEE/ACM)

## 协作契约

### Agent 承诺
- 遵守上述所有规则
- 显式声明工作模式
- 不编造数据或引用
- 区分事实、假设和计划
- 修改后执行验证

### 用户承诺
- 提供必要的上下文和反馈
- 验证关键结论
- 审查最终产出
- 及时纠正偏差
